# Agro Bot Flow Change

## Scope

- Применяй skill для изменений в `bot/`, `worker/` (если связано с бот-флоу), `locales/` и смежных API-клиентах бота.
- Не применяй для чисто backend-рефакторинга без изменения bot-flow.

## Workflow

1. Открывай `tasks.md` перед началом и соблюдай приоритеты; при прямом запросе пользователя выполняй запрос с сохранением проверок.
2. Определи затронутые точки входа:
- Команды (`bot/commands.js`, `bot/adminCommands.js`, `bot/planCommands.js`).
- Callback-ветки (`bot/callbacks/*`, `bot/flow/*`, `bot/planFlow.js`).
- Форматтеры (`bot/messageFormatters/*`) и тексты (`locales/ru.json`).
3. Держи консистентность UX:
- Добавляй дружелюбные ошибки и action-кнопки.
- Сохраняй пошаговый сценарий без лишних переходов.
- Для мультифото-диагноза соблюдай лимит 3-8 фото и обязательные кадры (общий вид, лист лицевая, лист изнанка), если задача касается этого flow.
4. Синхронизируй контракты и документацию при необходимости:
- `docs/srs.md`
- `docs/data_contract.md`
- профильные flow-доки (`docs/payment_flow.md`, `docs/prd.md`)
- `CHANGELOG.md`
5. Запускай проверки:
- `npm test --prefix bot`
- `./venv/bin/ruff check app tests` и `./venv/bin/python -m pytest`, если изменения затронули backend-контракт/эндпойнты
6. Формируй финальный отчёт:
- Что изменено в flow.
- Какие команды проверки выполнены.
- Что не запускалось и почему.

## Guardrails

- Не меняй ключи локализации частично: обновляй все связанные ветки/сообщения.
- Не создавай новые команды без проверки меню и fallback-сценариев.
- Не оставляй неиспользуемые callback-ветки после рефакторинга.
